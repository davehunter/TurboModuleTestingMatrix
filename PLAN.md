# TurboModuleTestingMatrix — Implementation Plan

## Context

The sandbox at `/Users/dave/src/TurboModuleTestingSandbox/` contains three sibling repos:

- **`TurboModuleTesting/`** — a CMake-based framework that lets a developer test React Native C++ TurboModules natively on macOS, by reusing the CocoaPods artifacts of a "host" RN app. Its public API (`TurboModuleTesting_ConfigureBasedOnApp(app_path)`, `TurboModuleTesting_AddTurboModuleDependencies(target, name)`) is RN-version-agnostic: it discovers RN headers, Hermes, third-party pods, and codegen output via existence checks. It already adapts to known path conventions for both RN 0.81.5 and 0.83.1.
- **`TurboModuleTestingExample/`** — a reference consumer: a host app (RN 0.83.1) plus an `RTNTestableModule` (C++ TurboModule with TS spec, codegen config, podspec, Google Test cases) plus a root `CMakeLists.txt` that fetches the framework via `FetchContent` and wires up tests.
- **`TurboModuleTestingMatrix/`** — currently empty. We're building it.

**Problem.** The framework is informally tested against two RN versions. It needs a continuously-runnable matrix that proves it works across many RN versions, so framework changes can be validated against every supported version before release. Adding a new supported RN version must be trivial: ideally one entry in a config file plus running one script.

**Outcome.** A repo that (a) holds a vendored test module, (b) holds one committed, fully-locked host app per RN version, (c) generates a new version's host app via a single script, (d) runs the framework's tests against every (or one) version with per-phase per-version reporting, and (e) is invoked by a GitHub Actions workflow on PRs against `TurboModuleTesting`.

## Locked decisions

1. **One folder per RN version**, committed, generated by a script (not a hand-maintained template).
2. **Generation** uses `npx @react-native-community/cli@<matching> init` so each version's host app reflects what RN actually produces for that version.
3. **Framework source**: local sibling `../TurboModuleTesting` by default; override via `TURBO_MODULE_TESTING_SRC` (local path) or `TURBO_MODULE_TESTING_TAG` (GitHub tag).
4. **Runner**: local macOS AND GitHub Actions. CI is triggered from `TurboModuleTesting/.github/workflows/pr.yml`, which checks out the Matrix and runs it against the PR's framework checkout.
5. **Reporting**: per-phase per-version (npm install / pod install / codegen check / cmake configure / build / ctest). Failure in one phase short-circuits subsequent phases for that version but not for the matrix.
6. **Lock files committed**: `package-lock.json`, `Podfile.lock`, `Gemfile.lock`. Gitignored: `node_modules/`, `ios/Pods/`, `ios/build/`, `vendor/bundle/`, `build/`, `results/`.

## Version registry — `versions.json` (§2)

```json
{
  "versions": [
    {
      "rn": "0.81.5",
      "cli": "@react-native-community/cli@15",
      "cliInitFlavor": "community",
      "node": ">=20",
      "ruby": "3.2",
      "notes": "Uses Pods/ReactNativeDependencies/Headers wrapper."
    },
    {
      "rn": "0.83.1",
      "cli": "@react-native-community/cli@20",
      "cliInitFlavor": "community",
      "node": ">=20",
      "ruby": "3.2",
      "notes": "Reference version from the Example repo."
    }
  ]
}
```

- `rn` — full RN version; must match `apps/<rn>/` and (if present) `overlays/<rn>/`.
- `cli` — npm-resolvable spec for `npx ... init`.
- `cliInitFlavor` — escape hatch (`"community"` today; future RN init mechanisms get a new flavor without special-casing existing versions).
- `node`, `ruby`, `notes` — informational; surfaced in the report header.

**Per-version overrides go in `overlays/<rn>/`, not in JSON.** Every path under `overlays/<rn>/` is rsynced over the generated app after `init`. `overlays/_shared/` applies first; then `overlays/<rn>/`. A special file `package.json.patch.json` is applied as a JSON merge patch via `jq` (this is how we inject `"rtn-testable-module": "file:../../../RTNTestableModule"` into every version's `package.json`).

## `scripts/generate.sh <version> [--force|--update]` (§3)

End-to-end behavior, in order:

1. Parse `version`; look up its row in `versions.json` via `jq`. Bail clearly if unknown.
2. **Idempotency.** If `apps/<v>/HostApp` exists:
   - default: error with instructions for `--force` or `--update`;
   - `--force`: `rm -rf` and re-init;
   - `--update`: skip `init`, re-apply overlays, re-run `npm install` + `bundle install` + `pod install`. This is the dev-common case after a shared overlay or `RTNTestableModule/` change.
3. **`init` into a tempdir**, then move to `apps/<v>/HostApp`:
   ```
   npx --yes ${cli} init HostApp --version ${version} --pm npm --skip-install --install-pods false --replace-directory true
   ```
   Flags chosen so we can layer overlays before any install.
4. **Apply overlays.** `rsync -a overlays/_shared/` then `rsync -a overlays/${version}/` (if it exists). Apply `package.json.patch.json` files (shared first, then per-version) via `jq -s '.[0] * .[1]'`.
5. **Install JS deps**: `npm install` → produces `package-lock.json` (committed) and `node_modules/` (gitignored).
6. **Install Ruby deps**: `bundle config set --local path 'vendor/bundle' && bundle install` → produces `Gemfile.lock` (committed) and `vendor/bundle/` (gitignored).
7. **Install Pods**: `bundle exec pod install`. For RN 0.81+, `prepare_react_native_project!` in the Podfile triggers codegen as part of pod install — that's what populates `ios/build/generated/ios/.../NativeRTNTestableModuleJSI.h` (the file the framework's `TurboModuleTesting_AddTurboModuleDependencies` discovers).
8. **Codegen sanity check.** Verify `NativeRTNTestableModuleJSI.h` is present at one of the two known codegen base paths (`ios/build/generated/ios/ReactCodegen/` or `ios/build/generated/ios/`). If absent, fall back to `npx react-native codegen` and re-check; otherwise fail with a precise error naming the searched paths.
9. **Print next steps**: tell the developer to `git add apps/<v>/` and which paths are expected to be committed vs gitignored.
10. **Exit non-zero on any failure.** Generation is the authoring step; it must be clean.

## `scripts/run-matrix.sh [version...]` (§4)

- Zero args → run every version in `versions.json`. Args → run only those (validated).
- Resolves framework source: `TURBO_MODULE_TESTING_SRC` (path) → `TURBO_MODULE_TESTING_TAG` (GitHub tag) → fallback to `../TurboModuleTesting` if present → fail.
- `MATRIX_RUN_ID` default: `$(date -u +%Y%m%dT%H%M%SZ)` plus short sha.

For each version, run phases sequentially:

| Phase | Command | Cwd |
|---|---|---|
| `npm-install` | `npm ci` (fallback `npm install` with warning) | `apps/<v>/HostApp` |
| `pod-install` | `bundle install && bundle exec pod install` | `apps/<v>/HostApp/ios` |
| `codegen-check` | Verify `NativeRTNTestableModuleJSI.h` exists; if not, `npx react-native codegen` | `apps/<v>/HostApp` |
| `cmake-configure` | `cmake -S . -B build/<v> -G Ninja -DEXAMPLE_APP_PATH=apps/<v>/HostApp` | matrix root |
| `cmake-build` | `cmake --build build/<v>` | matrix root |
| `ctest` | `ctest --test-dir build/<v> --output-on-failure --output-junit results/<run>/<v>/junit.xml` | matrix root |

`scripts/lib/common.sh` wraps each phase: records `start_iso`, `end_iso`, `duration_ms`, `exit_code`, `status` (`pass`/`fail`/`skipped`/`timeout`); streams output to terminal (prefixed `[<v>][<phase>]`) and to `results/<run>/<v>/logs/<phase>.log`; per-phase wall-clock timeout.

Failure in a phase marks subsequent phases for that version as `skipped` but does not stop the matrix. After all versions, `scripts/lib/report.sh` writes:
- `results/<run>/summary.json` (machine-readable; cited in §4 plan output schema).
- `results/<run>/summary.txt` (fixed-width table printed to stdout).
- `results/latest` symlink updated atomically.

Exit code `0` iff every phase of every version is `pass`; otherwise `1`.

## Root `CMakeLists.txt` (§5)

Configured **once per version** by the driver (`-DEXAMPLE_APP_PATH=apps/<v>/HostApp -B build/<v>`). The CMake file itself does not iterate versions — keeping this concern in shell avoids `FetchContent` / cache / codegen-path discovery interactions across versions.

`TurboModuleTestingMatrix/CMakeLists.txt`:
```cmake
cmake_minimum_required(VERSION 3.22)
project(TurboModuleTestingMatrix LANGUAGES CXX)
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

if(NOT DEFINED EXAMPLE_APP_PATH)
  message(FATAL_ERROR "Pass -DEXAMPLE_APP_PATH=apps/<rn>/HostApp; matrix is driven one version at a time.")
endif()

enable_testing()
include("CMake/IncludeGoogleTest.cmake")
include("CMake/IncludeTurboModuleTesting.cmake")
include("CMake/AddTest.cmake")

include_turbo_module_testing("${EXAMPLE_APP_PATH}")
add_subdirectory(RTNTestableModule)
```

`TurboModuleTestingMatrix/CMake/IncludeTurboModuleTesting.cmake` extends the Example's version (see `/Users/dave/src/TurboModuleTestingSandbox/TurboModuleTestingExample/CMake/IncludeTurboModuleTesting.cmake:1-15`) to honor `TURBO_MODULE_TESTING_SRC` / `TURBO_MODULE_TESTING_TAG` env vars and to fall back to the sibling `../TurboModuleTesting` when neither is set. `FetchContent_Declare(... SOURCE_DIR ...)` is used for local paths.

`TurboModuleTestingMatrix/CMake/AddTest.cmake` and `IncludeGoogleTest.cmake` copied verbatim from `/Users/dave/src/TurboModuleTestingSandbox/TurboModuleTestingExample/CMake/`.

**Vendor it** — `cp -a /Users/dave/src/TurboModuleTestingSandbox/TurboModuleTestingExample/RTNTestableModule /Users/dave/src/TurboModuleTestingSandbox/TurboModuleTestingMatrix/RTNTestableModule` once, then the matrix owns the copy.

## GitHub Actions in `TurboModuleTesting/.github/workflows/pr.yml` (§7)

New file. Triggered on PR. Two jobs:

1. `build-matrix` (ubuntu-latest, cheap): checks out the Matrix at `main`, reads `versions.json`, emits the list as a workflow output for fan-out.
2. `matrix` (macos-15, one runner per RN version via `strategy.matrix`, `fail-fast: false`):
   - Checks out the framework PR at `path: TurboModuleTesting`.
   - Checks out the Matrix at `path: TurboModuleTestingMatrix` (with an optional `workflow_dispatch.inputs.matrix_ref` for co-development of framework + matrix changes).
   - Sets up Node 20, Ruby 3.2, selects Xcode 16, `brew install cmake ninja`.
   - Caches `apps/<v>/HostApp/node_modules` keyed by `package-lock.json`.
   - Caches `apps/<v>/HostApp/ios/Pods` and `vendor/bundle` keyed by `Podfile.lock` + `Gemfile.lock`.
   - Caches `build/<v>` keyed by framework cpp/cmake hashes + matrix CMake hashes.
   - Runs `./scripts/run-matrix.sh ${{ matrix.rn }}` with `TURBO_MODULE_TESTING_SRC=$GITHUB_WORKSPACE/TurboModuleTesting`.
   - Uploads `results/` as an artifact.
   - Publishes `junit.xml` via `mikepenz/action-junit-report@v4` for PR check UI.

Fan-out gives parallelism (PR latency > runner cost). Caches drop pod install from minutes to seconds on rerun.

## Open risks to confirm during implementation

1. **Codegen trigger reliability.** Pod install reliably runs codegen for RN 0.81+. If older versions are added later, may need to call `npx react-native codegen` explicitly. Documented in `versions.json` notes per version.
2. **`init` flag stability across CLI versions.** `--skip-install`, `--install-pods false`, `--replace-directory` are present in `@react-native-community/cli` 15–20. Older/newer CLIs may differ — the `cliInitFlavor` field is the escape hatch.
3. **RN 0.78+ CLI changes.** The community CLI was reduced around RN 0.78. Confirm `npx @react-native-community/cli@latest init --version <rn>` still works for current targets; if not, introduce a `"createApp"` flavor.
4. **Ruby / Xcode pinning per version.** Plan uses Ruby 3.2 and Xcode 16 uniformly. If a future older RN target requires different Ruby/Xcode, surface those via `versions.json` and select per CI job.
5. **`Matrix-Ref` co-development.** Add `workflow_dispatch.inputs.matrix_ref` from day one so PRs that need matrix changes can be co-tested without merging Matrix first.
